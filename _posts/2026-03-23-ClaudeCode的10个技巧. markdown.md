## 1 Do more in parallel

Do more in parallel

Spin up 3–5 git worktrees at once, each running its own Claude session in parallel. It's the single biggest productivity unlock, and the top tip from the team. Personally, I use multiple git checkouts, but most of the Claude Code team prefers worktrees -- it's the reason 
@amorriscode
 built native support for them into the Claude Desktop app!

Some people also name their worktrees and set up shell aliases (za, zb, zc) so they can hop between them in one keystroke. Others have a dedicated "analysis" worktree that's only for reading logs and running BigQuery

See https://code.claude.com/docs/en/common-workflows#run-parallel-claude-code-sessions-with-git-worktrees

并行做更多

同时启动 3 到 5 个 git 工作树，每个并行运行自己的 Claude 会话。这是最大的生产力解锁，也是团队最重要的建议。我个人使用多个 git 检查，但大多数 Claude Code 团队更喜欢 worktrees——这也是为什么 
 @amorriscode 
 Claude 桌面应用内置了原生支持！

有些人还会给工作树命名，设置 shell 别名（za、zb、zc），这样他们就能在一个键键之间切换。还有些有专门的“分析”工作树，专门用于读取日志和运行 BigQuery

参见  https://code.claude.com/docs/en/common-workflows#run-parallel-claude-code-sessions-with-git-worktrees

## 2 plan mode

Start every complex task in plan mode. Pour your energy into the plan so Claude can 1-shot the implementation.

One person has one Claude write the plan, then they spin up a second Claude to review it as a staff engineer. 

Another says the moment something goes sideways, they switch back to plan mode and re-plan. Don't keep pushing. They also explicitly tell Claude to enter plan mode for verification steps, not just for the build

以计划模式启动所有复杂任务。把精力都投入到计划中，让克劳德一枪就能完成这个实现。

一个人让一个 Claude 写计划，然后再启动另一个 Claude 作为工程师审核。

还有人说一旦出现问题，他们就会切回计划模式重新规划。别再逼了。他们还明确告诉 Claude 在验证步骤时进入计划模式，而不仅仅是构建阶段

## 3 Invest CLAUDE.md

Invest in your http://CLAUDE.md. After every correction, end with: "Update your http://CLAUDE.md so you don't make that mistake again." Claude is eerily good at writing rules for itself.

Ruthlessly edit your http://CLAUDE.md over time. Keep iterating until Claude's mistake rate measurably drops.

One engineer tells Claude to maintain a notes directory for every task/project, updated after every PR. They then point http://CLAUDE.md at it.

投资你的  http://CLAUDE.md .每次更正后，最后写道：“更新你的信息  http://CLAUDE.md  ，避免再犯同样的错误。”克劳德擅长为自己制定规则，出奇地好。

不断地进行编辑。  http://CLAUDE.md  不断迭代，直到克劳德的错误率明显下降。

一位工程师告诉 Claude 为每个任务/项目维护一个笔记目录，每次 PR 后更新。然后他们指  http://CLAUDE.md  着它。

## 5 Just say "fix"

Claude fixes most bugs by itself. Here's how we do it:

Enable the Slack MCP, then paste a Slack bug thread into Claude and just say "fix." Zero context switching required.

Or, just say "Go fix the failing CI tests." Don't micromanage how.

Point Claude at docker logs to troubleshoot distributed systems -- it's surprisingly capable at this.

Claude 自己修复了大多数 bug。我们的做法如下：

启用 Slack MCP，然后把 Slack 的错误帖子粘贴到 Claude 里，只说“fix”。完全不需要切换上下文。

或者，干脆说“去修正那些失败的人工认知测试”。不要事无巨细地管理怎么做。

让 Claude 去看 docker 日志来排查分布式系统——它在这方面出乎意料地有能力。

## 6 Level up your prompting

Level up your prompting

a. Challenge Claude. Say "Grill me on these changes and don't make a PR until I pass your test." Make Claude be your reviewer.  Or, say "Prove to me this works" and have Claude diff behavior between main and your feature branch

b. After a mediocre fix, say: "Knowing everything you know now, scrap this and implement the elegant solution"

c. Write detailed specs and reduce ambiguity before handing work off. The more specific you are, the better the output

提升你的提示能力

a. 挑战克劳德。说“你就这些改动好好问我，别再创个人纪录，等我通过你的测试。”让克劳德成为你的评审人。 或者，说“证明给我看这可行”，然后用 Claude Diff 在主分支和你的功能分支之间做行为差异

b.在一个平庸的修复后，可以说：“现在知道了所有知识，放弃这个，实现优雅的解决方案。”

c. 在交接工作前撰写详细规格并减少歧义。你越具体，输出越好

## 7 Terminal & Environment Setup

Terminal & Environment Setup

The team loves Ghostty! Multiple people like its synchronized rendering, 24-bit color, and proper unicode support.

For easier Claude-juggling, use /statusline to customize your status bar to always show context usage and current git branch. Many of us also color-code and name our terminal tabs, sometimes using tmux — one tab per task/worktree. 

Use voice dictation. You speak 3x faster than you type, and your prompts get way more detailed as a result. (hit fn x2 on macOS)

More tips: https://code.claude.com/docs/en/terminal-config

终端与环境设置

团队都喜欢 Ghostty！很多人喜欢它的同步渲染、24 位色彩和合理的 Unicode 支持。

为了更方便地管理 Claude，可以用 /statusline 自定义状态栏，始终显示上下文使用情况和当前 git 分支。我们许多人还会给终端标签页做颜色编码和命名，有时用 tmux 来命名——每个任务/工作树一个标签页。

使用语音输入。你说话的速度是打字的三倍，提示词也因此变得更详细。（在 macOS 上按 fn x2）

更多建议： https://code.claude.com/docs/en/terminal-config

## 8 Use subagents

Use subagents

a. Append "use subagents" to any request where you want Claude to throw more compute at the problem

b. Offload individual tasks to subagents to keep your main agent's context window clean and focused

c. Route permission requests to Opus 4.5 via a hook — let it scan for attacks and auto-approve the safe ones (see https://code.claude.com/docs/en/hooks#permissionrequest)

使用子代理

a.在你希望 Claude 对问题投入更多计算时，在请求后加上“使用子代理”。

b.将单个任务分给子代理，以保持主代理的上下文窗口干净且聚焦

c. 通过钩子将权限请求路由到 Opus 4.5 ——让它扫描攻击并自动批准安全攻击（见  https://code.claude.com/docs/en/hooks#permissionrequest ）

## 9 Use Claude for data & analytics

Use Claude for data & analytics

Ask Claude Code to use the "bq" CLI to pull and analyze metrics on the fly. We have a BigQuery skill checked into the codebase, and everyone on the team uses it for anlytics queries directly in Claude Code. Personally, I haven't written a line of SQL in 6+ months.

This works for any database that has a CLI, MCP, or API.

使用 Claude 进行数据和分析

让 Claude Code 使用“bq”CLI 实时拉取和分析指标。我们已经把 BigQuery 技能录入了代码库，团队里每个人都直接用 Claude Code 做分析查询。就我个人而言，我已经有 6+个月没写过一行 SQL 了。

这适用于任何带有 CLI、MCP 或 API 的数据库。

## 10 Learning with Claude

Learning with Claude

A few tips from the team to use Claude Code for learning:

a. Enable the "Explanatory" or "Learning" output style in /config to have Claude explain the *why* behind its changes

b. Have Claude generate a visual HTML presentation explaining unfamiliar code. It makes surprisingly good slides!

c. Ask Claude to draw ASCII diagrams of new protocols and codebases to help you understand them

d. Build a spaced-repetition learning skill: you explain your understanding, Claude asks follow-ups to fill gaps, stores the result

与克劳德一起学习

团队给出的一些使用 Claude Code 学习的小贴士：

a.在 /config 中启用“解释”或“学习”输出样式，让 Claude 解释其更改背后的*原因*。

b.让 Claude 生成一个可视化的 HTML 演示文稿，解释不熟悉的代码。它做出的幻灯片出乎意料地好！

c. 让 Claude 绘制新协议和代码库的 ASCII 图，帮助你理解它们

d. 建立间隔重复学习技能：你解释你的理解，Claude 要求后续补问以填补空白，并存储结果

## 11 Use output styles

Use output styles

Run /config and set an output style to have Claude respond using a different tone or format.

We recommend enabling the "explanatory" output style when getting familiar with a new codebase, to have Claude explain frameworks and code patterns as it works.

Or use the "learning" output style to have Claude coach you through making code changes.

You can also create custom output styles to adjust Claude's voice the way you like.

https://code.claude.com/docs/en/output-styles

使用输出样式

运行 /config，设置输出样式，让 Claude 用不同的语气或格式来响应。

我们建议在熟悉新代码库时启用“解释性”输出风格，让 Claude 在运行时解释框架和代码模式。

或者用“学习”输出方式让 Claude 指导你修改代码。

你也可以自定义输出样式，根据喜好调整 Claude 的声音。

 https://code.claude.com/docs/en/output-styles

## 12 Customize all the things

Customize all the things!

Claude Code is built to work great out of the box. When you do customize, check your settings.json into git so your team can benefit, too. We support configuring for your codebase, for a sub-folder, for just yourself, or via enterprise-wide policies.

Pick a behavior, and it is likely that you can configure it. We support 37 settings and 84 env vars (use the "env" field in your settings.json to avoid wrapper scripts).

Learn more:

自定义所有东西！

Claude Code 开箱即用，设计得非常好。自定义时，把 settings.json 导入 git，这样团队也能受益。我们支持为您的代码库、子文件夹、仅个人配置，或通过企业范围的策略配置。

选择一种行为，你很可能可以配置它。我们支持 37 个设置和 84 个环境变量（请在 settings.json 中使用“env”字段以避免封装脚本）。
